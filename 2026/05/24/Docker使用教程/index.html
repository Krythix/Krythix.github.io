一, 什么是Docker

为什么需要Docker

在软件开发中, 经常遇到一个问题: “明明在我机器上能跑, 为什么到你那里就不行了?”

原因很简单 — 每个人的机器环境不同: 操作系统不同、软件版本不同、依赖库不同、配置文件不同。传统做法是手动配置环境, 容易出错且不可复现。

Docker 解决的就是这个问题: 把应用和它所依赖的一切(代码、运行时、系统工具、系统库、配置)打包在一起, 形成一个标准化的、可移植的单元。

Docker 是一个开源的容器化平台, 让你可以将应用程序及其所有依赖打包到一个轻量级、可移植的容器中, 然后在任何支持Docker的环境中运行。

容器 vs 虚拟机

理解容器最好的方式是把它和虚拟机(VM)对比:

虚拟机 Docker容器
虚拟化层次 硬件层 操作系统层
启动速度 分钟级 秒级
资源占用 每个VM有完整OS, 几GB 共享宿主机内核, 几十MB
隔离性 完全隔离 进程级隔离
镜像大小 通常几GB 通常几十到几百MB

简单来说: 虚拟机虚拟的是硬件, 容器虚拟的是操作系统。 Docker容器共享宿主机的操作系统内核, 所以轻量得多。

二, Docker核心概念

Docker 有三个最核心的概念, 理解它们就理解了 Docker 的根基:

1. 镜像 (Image)

镜像是容器的蓝图, 是一个只读模板。

类似于虚拟机快照或操作系统的ISO文件。镜像包含了运行应用程序所需的一切。镜像是分层构建的, 每一层都是只读的, 这种设计让镜像复用变得非常高效。

类比: 镜像 = 类的定义 (Class)

2. 容器 (Container)

容器是镜像的运行实例。

你可以从同一个镜像创建、启动、停止、删除多个容器。每个容器是相互隔离的, 有自己的文件系统、进程空间和网络。

类比: 容器 = 类的实例对象 (Object)

3. 仓库 (Registry / Repository)

仓库是存放镜像的地方。

  • Docker Hub 是 Docker 官方的公共镜像仓库 (类似 GitHub)
  • 你可以从仓库**拉取(pull)**镜像到本地
  • 也可以把自建的镜像**推送(push)**到仓库
  • 一个仓库可以有多个版本的镜像, 用**标签(tag)**区分

三者关系

4. Dockerfile

Dockerfile 是一个文本文件, 包含构建镜像的指令。

用 Dockerfile 可以自动化构建镜像, 保证每次构建的结果一致。这是”基础设施即代码(Infrastructure as Code)”思想的体现。

5. Docker Compose

Docker Compose 用于定义和管理多容器应用。

当应用由多个服务组成时(如 web + 数据库 + 缓存), 使用 Compose 可以在一个文件中定义所有服务, 一条命令启动全部。

五, 容器操作

容器是镜像的运行实例, 也是我们实际使用 Docker 的核心。

创建并启动容器 (docker run)

docker run 是最重要的命令, 它 = 创建 + 启动容器。

参数解析:

  • -d (detach): 后台运行容器
  • -p 8080:80 (publish): 端口映射, 将宿主机8080端口映射到容器80端口
  • --name mynginx: 给容器命名(不指定则随机生成名字)
  • nginx: 使用的镜像名

六, Dockerfile — 构建自己的镜像

Dockerfile 是构建镜像的”配方”。

基本语法

Dockerfile 由一系列指令组成, 每条指令创建一个新的镜像层:

指令详解

FROM — 选择基础镜像

选择基础镜像的原则:

  • 能用 Alpine 就别用完整版 (体积小, 安全)
  • 有官方镜像就用官方镜像
  • 选择明确版本号, 不要依赖 latest

RUN — 构建时执行的命令

为什么要把多条命令用 && 串联? 每条 RUN 会创建一个新的镜像层。把相关命令合并在一个 RUN 中, 可以减少镜像层数, 减小镜像体积。

COPY vs ADD

优先使用 COPY: ADD 的自动解压和 URL 下载功能可能引发意外行为。除非明确需要自动解压, 一律用 COPY。

CMD vs ENTRYPOINT

组合使用的经典模式:

  • ENTRYPOINT 定义”做什么”
  • CMD 定义”默认参数”
  • docker run 后面的参数会替换 CMD

构建镜像

构建上下文: docker build 末尾的 . 是构建上下文目录。Docker daemon 会将这个目录下的所有文件发送给 Docker daemon 用来构建。因此不要把 Dockerfile 放在有很多文件的大目录下。

.dockerignore

在 Dockerfile 同级目录创建 .dockerignore 文件, 排除不需要发送到 Docker daemon 的文件:

多阶段构建 (Multi-stage Build)

多阶段构建可以在一个 Dockerfile 中使用多个 FROM, 实现”在第一个阶段编译, 在第二个阶段只复制编译结果”, 从而大幅减小最终镜像体积。

对比: 如果把所有编译工具都放在最终镜像里, 镜像可能有 800MB+。使用多阶段构建后, 最终镜像只有 Alpine 的 5MB + 编译出的二进制文件 ≈ 15MB。

七, 数据管理

容器是无状态的 — 删除容器后, 容器内的数据也会消失。要持久化数据, 需要用到 Docker 的存储机制。

三种数据存储方式

方式 数据存储位置 用途
Volume Docker管理的宿主机目录 (/var/lib/docker/volumes/) 持久化数据, 容器间共享
Bind Mount 宿主机上任意目录 开发时同步代码, 共享配置文件
tmpfs Mount 内存中 临时数据, 不想持久化

Volume (卷) — 推荐方式

Volume 由 Docker 管理, 与宿主机文件系统解耦。

-v vs --mount: -v 更简洁, --mount 更明确。官方推荐 --mount 用于新项目。

Bind Mount (绑定挂载)

将宿主机上的任意目录挂载到容器中。

典型场景: 开发时将本地代码目录挂载到容器, 修改代码后立即生效(配合热重载)。

十二, 最佳实践总结

  1. 一个容器一个进程: 每个容器只运行一个主要进程, 不要在一个容器里跑多个服务 (用 Compose 编排多服务)
  2. 使用 .dockerignore: 排除不需要的文件, 加快构建速度, 减小镜像体积
  3. 选择最小基础镜像: 优先使用 Alpine 版本的镜像
  4. 合并 RUN 指令: 减少镜像层数
  5. 多阶段构建: 分离构建环境和运行环境
  6. 不要以 root 运行: 在 Dockerfile 中使用 USER 切换用户
  7. 用 Volume 持久化数据: 不要依赖容器内的文件系统
  8. 给镜像打明确版本标签: 不要只用 latest
  9. 用 Compose 管理多容器应用: 不要在脚本里写一堆 docker run
  10. 定期清理: 定期执行 docker system prune 回收磁盘空间
  11. 扫描镜像漏洞: 使用 docker scout quickview <image> 检查安全漏洞
  12. 不要在镜像中存储密钥: 使用 Docker secrets 或环境变量传入
