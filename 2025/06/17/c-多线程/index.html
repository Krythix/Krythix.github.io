二, promise和future

完成单次的异步请求和响应

promise用来发送数据, future从promise中获得用来接受数据.

一个promise对应一个future

  1. promise

    • template<typename Res> class promise;
    • template<typename Res> class promise<Res&>; 两个线程传引用
    • template<> class promise<void>; 表示完成信号(set_value设置完成状态)
    • get_future() 返回一个future对象
    • set_value 设置值, 并把状态设置为就绪

  2. future

    • get 阻塞线程, 就绪时读取值
    • wait 等待就绪
    • wait_for 等待时长
    • wait_until 等到指定时刻
    • share

    future的拷贝函数被删除, 只能移动或引用

三, 互斥和锁

std::mutex

防止两个线程同时访问数据(非原子), 导致数据有错误.

  • lock 阻塞, 并锁定
  • try_lock 不阻塞, 不能锁定时返回false, 成功锁定并返回true
  • unlock

lock_guard

使用RAII管理一个锁, 方便管理, 异常安全

是一个模板, 接受一个可加锁的类型

lock_guard对象在构造时加锁, 析构时解锁

unique_lock

比lock_guard更方便, 可以自己加锁解锁的锁

  • unique_lock(mutex, std::defer_lock)延迟加锁(.lock), 解锁(.unlock)

timed_mutex

  • .lock

  • .try_lock_for 阻塞一定时间并返回ture/false

四, 读写锁(共享锁)

用普通锁读取, 写资源:

读时间比写时间短, 在写时, 读线程就都已经开始申请加锁, 写线程解锁后, 每个读线程依次读取, 效率低

想要让读线程可以同时读取数据

读写锁和互斥量shared_mutex

两种模式:

  • 独占模式(写): 同一时间只有一个线程可获得锁
    • .lock()
    • .try_lock()
    • .unlock()
    • RAII使用锁: unique_lock和lock_guard
  • 共享模式(读): 多个线程可共享这个锁
    • .lock_shared
    • .try_lock_shared
    • .unlock_shared
    • RAII使用锁: shared_lock

有了写锁, 其他所有锁都不能再加, 有了读锁, 只能继续加读锁 (在没有写锁申请时).

当一个线程持有读锁时，其他线程可以继续加读锁，但如果有线程正在等待写锁，后续读锁的申请会被阻塞, 防止写线程一直无法加锁

.wait

基础用法和问题

void wait(std::unique_lock<std::mutex>& lock);

只传一个unique_lock为参数时, 执行下面步骤

  1. 给lock解锁
  2. 阻塞等待其他线程调用.notify
  3. 给lock加锁

但是可能有丢失唤醒, 虚假唤醒

丢失唤醒: 在开始wait之前就有线程发送了唤醒

虚假唤醒: 因为操作系统, 可能有错误的唤醒信号

改进的使用方法

template <class Predicate> void wait(std::unique_lock<std::mutex>& lock, Predicate pred);

相当于(简易版):

丢失唤醒时, 因为pred已经被设置, 不会进入循环

虚假唤醒时, 因为pred没有被写线程设置, 所以还会继续while循环

一些注意事项

(1) 互斥锁的作用

如果发生这个情况:

线程会一直wait吗?

不会, 因为有互斥锁.

如果写线程在写数据, 那么锁在写线程中, 读线程不可能执行到判断pred的语句

如果读线程加了锁, 此时写线程在阻塞中

(2) 一些顺序

  1. wait 之前应该用unique_lock加锁
  2. 写线程中先设置数据和pred(pred可以通过数据推导), 再发送notify
